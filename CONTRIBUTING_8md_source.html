15 ## Commit Messages
16 
17 "Re-establishing the context of a piece of code is wasteful. We can't avoid it completely, so our
18 efforts should go to reducing it [as much] as possible. Commit messages can do exactly that and as a
19 result, a *commit message shows whether a developer is a good collaborator*." -- Peter Hutterer
20 
21 The seven rules of a great git commit message:
22 
23 1. separate subject from body with a blank line;
24 2. limit the subject line to 50 characters;
25 3. capitalise the subject line;
26 4. do not end the subject line with a period;
27 5. use the imperative mood in the subject line;
28 6. wrap the body at 72 characters;
29 7. use the body to explain _what_ and _why_ vs. _how_.
30 
31 See [How to Write a Git Commit Message](http://chris.beams.io/posts/git-commit/) by Chris Beams.

44 ## Coding Style
45 
46 Please follow [CppCoreGuidelines](https://github.com/isocpp/CppCoreGuidelines) for anything not
47 covered in this section.
48 
49 ### Filenames
50 
51 - [CamelCase](https://en.wikipedia.org/wiki/CamelCase) filenames;
52 - header files have `.hpp` extension;
53 - source files have `.cpp` extension;
54 - test files have `.ut.cpp` extension.
55 
56 ### Formatting
57 
58 - spaces instead of tabs;
59 - indent by 4 spaces;
60 - lines should not exceed 100 characters.
61 
62 [ClangFormat](http://clang.llvm.org/docs/ClangFormat.html) should be used to format source-code
63 changes. The
64 [GitClangFormat](https://github.com/llvm-mirror/clang/blob/master/tools/clang-format/git-clang-format)
65 script integrates [ClangFormat](http://clang.llvm.org/docs/ClangFormat.html) with
66 [Git](https://git-scm.com/).
67 
68 ### Naming
69 
70 - [CamelCase](https://en.wikipedia.org/wiki/CamelCase) for types and integer constants;
71 - [snake\_case](https://en.wikipedia.org/wiki/Snake_case) for functions, variables and non-integer
72  constants;
73 - types and integer constants start with an upper-case character;
74 - trailing underscore for private data-members.
75 
76 ### Class Layout
77 
78 The `public` section is first, followed by `protected` and then `private`. Within each section,
79 members are generally ordered as follows:
80 
81 - big six;
82 - const functions;
83 - non-const functions;
84 - data.
85 
86 Const functions precede non-const functions for two reasons:
87 
88 1. it is the natural order imposed when a mutable interface extends an immutable interface;
89 2. mutators will most likely be placed alongside the private data-member section, which is generally
90  more useful for maintainers.
91 
92 ### Assertions
93 
94 Use `assert` statements to formalise contract and document assumptions.
